La sélection des permissions dépend du périmètre décidé dans le cadre du projet. Par exemple, si les rôles que l'on souhaite créer ciblent la finance, alors le choix va se porter sur les permissions des applications financières.

Réduire le nombre de permission est important pour ne pas afficher une trop grande quantité d'information dans le tableau croisé dynamique. Cela nuierait à la visibilité par le responsable métier. Les applications comportant un très grand nombre de permissions nécessite peut-être en amont une phase de Role Mining applicatif afin de créer des profils plus fonctionnels.

Un point très important concernant les permissions est de s'assurer qu'elles sont compréhensibles par les métiers. La présence d'une description est un bon indicateur. Si les métiers ne comprennent pas la signification des permissions, la construction des rôles sera extrèmement difficile voire impossible. La qualité de la documentation des permissions est à apprécier dans le contexte client. En effet, des codes de permissions cryptiques ne sont pas nécessairement problématiques si ceux-ci sont couramment manipulés par les métiers.

Les permissions peuvent être documentées lors des ateliers mais la bonne pratique est de les documenter avant les ateliers et ceci pour deux raisons :

  • Documenter les permissions est un travail à effectuer avec le responsable applicatif et non le responsable métier
  • Le temps que consacre le responsable métier à un atelier est précieux. Il ne faut pas perdre de temps à se poser des questions sur la signification des permissions pendant les ateliers. Pour que la session soit efficace, il faut que le responsable métier puisse se concentrer sur les rôles.
