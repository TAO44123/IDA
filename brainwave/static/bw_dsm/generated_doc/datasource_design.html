Introduction

The new Datasource Management add-on provides an unified platform for defining and configuring the extraction and collection of data into Brainwave from various sources, directly from the web portal.

This add-on introduces the concept of Datasource in order to normalize the configuration and processing of data to be collected inside Brainwave,

A Datasource is a combination of a data collector and a channel to obtain the data to be collected.
- A data collector in Brainwave sense is an add-on that processes a set of data files in a specified format to feed the Brainwave Identity Ledger data model. Examples of data collectors are ActiveDirectory, CyberArk , Shared Folders, Google Drive, etc.
- A channel is a way to obtain the data to be collected. It ranges from a live connection to an LDAP server, a script that is run to extract data, to a mere data file that is manually dropped in a given location.

In order for a datasource to be recognized by the Datasource Management add-on, you need to design a datasource descriptor.

Designing a Datasource Descriptor

A datasource descriptor provides information on the characteristics and capabilities of a datasource.

  • basic information ( type, id, category, name, description, family, etc.)
  • options information: list of option and configuration GUI
  • mapping information
  • addtional flags

The datasource descriptor is a special Page file that needs to be included in the application project, either through the facet of the datasource, or externally anywhere in the project.

Here is an example of a minimal datasource descriptor:

ds_activedirectory_file = Page {
    title:  "Microsoft Active Directory - file"
    tags: "bwds_connector" , "file" , "csv", "microsoft"
    description: "Collects all provided LDIF files, etc ..."
    icon: "bw_dsm/L/bw_activedirectory.png"
    large-icon: "bw_dsm/XL/bw_activedirectory.png"  
     metadata: (     
        fixedName -> "ad"  ,  
        category -> "infrastructure", 
        family -> "LDAP",
        collector -> "bw_activedirectory",
        destPathVar -> "ad_sourceFolder",
        filePattern -> "*.ldif",
        hasMapping -> False,
        multiFile -> True
     )
}

The characteristics of a datasource are provided either through standard attribute of a Page (eg. title, tags, icon, etc.) or through a list of metadata key -> value pairs.

Descriptor page attributes table

Attribute Mandatory? Description Example
id mandatory unique identifier for the datasource throughout the system. the id should follow the following naming pattern: **ds_{collector}_{channel}** ds_activedirectory_file
title * mandatory the label of the datasource as it will appear in the Datasource management GUI. The title should take the form {collector}-{channel}. title can be localized Microsoft Active Directory - file
tags mandatory keywords that describe the datasource, so that it can be searched. You must include the special bwds_connector tag so that the page is recognized as a connector descriptor “bwds_connector” , “file” , “csv”, “microsoft”
description mandatory description of the purpose of this datasource to be displayed in the configuration GUI. The desciption must be up to 5 lines to fit in the GUI. You can force newlines using \n. description can be localized.
icon mandatory 48x48 icon of the datasource , to be displayed in the wizard page. the icon must be located in /reports/icons folder of the project bw_dsm/L/bw_activedirectory.png
large-icon mandatory 96x96 large icon of the datasource , to be displayed in the wizard selection page. the icon must be located in /reports/icons folder of the project bw_dsm/XL/bw_activedirectory.png
metadata mandatory list of key-value pairs that describe addtional features of the datasource, in the form metadata: ( key1 -> value1, key2 -> value2, ... ). Metadata possible keys are described in the new table metadata: ( category -> "infrastructure", family -> "LDAP", multiFile -> True )

Descriptor page metadata attributes table

Attribute Mandatory? Description Example
category * mandatory category of the datasource. must be one of the following values: infrastructure, identity, application, cloud category -> “application”
family optional sub-category of the datasource LDAP
collector mandatory name of the collector facet. “bwdsm_generic_hr”
action optional name of the script to execute to extract the data, if available “activedirectory”
destPathVar mandatory name of the configuration variable that holds the location of imported files in the project “ad_sourceFolder”
hasDestDir optional flag indicating whether data files should be copied to a subfolder of import files. If set to true, the configuration UI will prompt for the sub folder name. Defaults to False
multiFile optional boolean flag indicating whether the collect accepts multiple files. must match the iterate over files silo flag. Default is False
filePattern mandatory file pattern of the data files. Same syntax as iterated silos "*.ldif"
hasMapping mandatory boolean flag indicating whether a mapping can be applied to the data file, enabling Mapping in the configuration UI. Defaults to False. when set to True, the imodel* metadata attributes must be also set
imodelcolumns optional semi-column separated list of expected model columns for the mapping “hrcode;givenname;surname;altname”
imodelcolumnsmandatory optional semi-column separated list of boolean values indicating whether a model column is mandatory “False;False;True”
imodelcolumnsdescription optional semi-column separated list of column descriptions for the model columns. Can be localized “Employee ID number;First name;Last name;Alternative or maiden name”
imodelcolumnsexample optional semi-column separated list of value examples for model columns “ID0000183;Denis;GEOFFRAY;JACOB;Denis GEOFFRAY”
imodelcolumnstype optional semi-column separated list of value types for the model column. Allowed types are: String, Boolean, Date; Number “String;Boolean;Date”

Designing Configuration Options UI

Some datasources may have additional user-configurable options. For example, a query-based datasource will have options for the URL and credentials to connect to, An application permission rights datasource options might include the name and description of the Application to create in the Identity Ledger.

In order to design configuration UI for your datasource, carry out the following steps: - create two services for serializing and deserializing the options - create a dynamic page fragment to hold the configuration UI. The fragment name must be exactly {page_name}_cfg.
In the above example, the ds_activedirectory_file datasource configuration fragment would be named ds_activedirectory_file_cfg.

Serialization services

Create two services for serializing and deserializing the options. Names do not matter, but they must be unique application wide.

Deserialization service: this service is in charge of transforming the JSON string into individual variables that will hold the option values. - the option names must be passed to the service as a comma separated list String. Order does not matter. We recommand to use a Constant so that the same value can be passed to both services - The service must have an Ouput parameter per option.

Serialization service: this service is in charge of transforming back the option variables into a JSON string. - the option names must be passed to the service as a comma separated list String into the varnames parameter. Order does not matter. We recommand to use a Constant so that the same value can be passed to both services. - the service must have an Input parameter per option. - if some options values need to be encrypted, add two parameters to your service: secretnames and secreddsname. you must then pass the options names to encrypt (as comma separated string) to the service into the secretnames parameter. You must also pass the name of the datasource (contained in the dsname variable) to the secreddsname parameter of the service. It will be used to build a key for storing the encrypted values in the vault.

Options UI Fragment

The option fragment is displayed in the datasource configuration wizard options page. The fragment is in charge of providing the configuration UI, serializing/deserializing the options values and making any required validation.

The fragment has 3 mandatory variables: - dsname : input variable will contain the id of the datasource. Used for encrypting secret options - edit : boolean variable. False if creating the datasource, True if editing an existing datasource - options : in/out string variable containing a JSON representation of the options.

Below is a code template for a datasource with 4 options ( ldapurl, domain, login and password) , one of which being encrypted (password). Copy and adapt this code to your own datasource

Datasource code template

ad_ldap_params = Constant { value: "login,password,domain,ldapurl" }

deserializeOptions_adp = EventService {
  javascript-file: "/webportal/pages/bw_dsm/cfg/ds_cfg.javascript"
  javascript-method: deserializeOptions
  varnames = Input { default: Constant ad_ldap_params }
  options = Input
  login = Output { }
  password = Output { }
  domain = Output { }  
  ldapurl = Output { }
}

serializeOptions_adp = StringService {
  javascript-file: "/webportal/pages/bw_dsm/cfg/ds_cfg.javascript"
  javascript-method: serializeOptions
  varnames = Input { default: Constant ad_ldap_params }
  secretnames = Input { default: "password"}
  secretdsname = Input { mandatory: True } /* name of the datasource, used for secret key*/
  login = Input { }
  password = Input { }
  domain = Input { } 
  ldapurl = Input { }
}

ds_activedirectory_ldap_cfg = PageFragment {
  post-render-event: ExecuteNamedAction doInit
  dsname = Variable  
  edit = Variable { type: Boolean } 
  options = Variable { type: String  }/* in/out json string for all options. value updated when doCommit is called */
  // custom
  ldapurl = Variable { initial: "ldap://"}
  login = Variable
  password = Variable
  domain = Variable
  
   doInit = NamedAction(
     Call deserializeOptions_adp ( options to options, ldapurl from ldapurl, login from login, password from password, domain from domain)
   )
   
   doCommit = NamedAction(
      Set @serializeOptions_adp ( dsname to secretdsname, 
          ldapurl to ldapurl, login to login, password to password, domain to domain   ) to options
   )
   
  Edit { 
     label: "URL"
     variable: ldapurl
     match: "ldap:\\/\\/.+"
     not-empty:True
  }
  Edit { 
     label: "Login"
     variable: login
     not-empty:True
  }
  Edit { 
     label: "Password"
     variable: password
     not-empty:True
     password-mode: True
  }
  Edit { 
     label: "Domain"
     variable: domain
     not-empty:True
     tooltip: "Name of the domain to create in the Identity Ledger"
  } 
}

Enabling column mapping

To enable column mapping for your datasource, you need to carry out the following steps: - set hasMapping to True in the datasource metadata descriptor - set imodelcolumns , imodelcolumnsmandatory , imodelcolumnstype, imodelcolumnsdescription and imodelcolumnsexample to semi-column separated list of values. All 5 lists must have the same cardinality.

Here is a example code of the descriptor page for a datasource with mapping enabled:

ds_generic_hr_csv = Page {
    title: $ds_generic_hr_csv_NLS.title
    tags: "bwds_connector" 
    description: $ds_generic_hr_csv_NLS.description
    icon: "bw_dsm/L/bw_genericapp.png"
    large-icon: "bw_dsm/XL/bw_genericapp.png"
    metadata: (  
     category -> "identity", 
     collector -> "bwdsm_generic_hr",
     destPathVar -> "bw_generic_hr_sourceFolder", 
     filePattern -> "*.csv",
     multiFile -> False  ,
     hasMapping -> True,
    imodelcolumns -> "hrcode;givenname;surname;altname;fullname",
    imodelcolumnsmandatory ->"False;False;False;False;True",
     imodelcolumnsdescription -> "Employee ID number;First name;Last name;Alternative or maiden name;Complete name",
     imodelcolumnsexample -> "ID0000183;Denis;GEOFFRAY;JACOB;Denis GEOFFRAY",
     imodelcolumnstype -> "String;String;String;String;String"
   )    
}
